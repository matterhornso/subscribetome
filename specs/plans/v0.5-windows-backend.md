# Plan — Workstream B: Windows Credential Manager backend

**Target version:** v0.5.0
**Spec parent:** [`specs/cross-platform-and-codex.md`](../cross-platform-and-codex.md) §5 (Windows row of the backend table)
**Status:** planned, unstarted
**Estimated build:** half a day of focused work (the seam already exists from v0.3.1; Workstream B is one new file + tests + the resolver wiring)

## 1. Goal

A user on Windows runs `stm dashboard`, adds a key, opens a Claude Code or
Codex session — the key is stored in Windows Credential Manager (DPAPI-
encrypted, per-user) and resolves back the same way the macOS / Linux
backends do today. The active backend is shown in `stm status` and the
dashboard. No silent fallback to plaintext.

After this ships, **stm covers all three desktop OSes** for the v1
"strong" guarantee (per-command rewrite via Claude Code, or session-env
via Codex). Headless Linux remains the only desktop hole.

## 2. Why this is the right next OS

- Single new file plugs into the `KeyStore` resolver from v0.3.1
  (`src/keystores/index.ts`). The resolver, the dashboard pill, `stm
  status`, the `STM_KEYSTORE` override aliases — all of them work as-is.
- Windows Credential Manager is the documented, supported, DPAPI-backed
  store. Per-user, no daemon needed, available on every consumer + Pro
  + Server SKU.
- Reach: a meaningful fraction of Claude Code users are on Windows
  natively (not WSL). Today they get "platform unsupported" from the
  resolver.

## 3. Why the implementation is non-trivial despite the seam

Windows has **no portable CLI equivalent to `secret-tool`**:
- `cmdkey` (built-in) can write + delete but **cannot read** a password
  back. Useless for `get`.
- `wincred` (Windows Credential API) is the only way to read. It's a
  Win32 API, not a CLI.
- `keytar` (npm) is archived (2026-03), no maintained replacement
  ships standalone.

So we need **Bun FFI** to call `CredWriteW` / `CredReadW` / `CredDeleteW`
in `advapi32.dll` directly. This is the meaningful engineering work —
not the resolver wiring.

## 4. Files to add / change

### 4.1 New file: `src/keystores/windows-credential.ts`

The Windows backend. Shape exactly matches `src/keystores/mac.ts` and
`src/keystores/linux-secret-service.ts`:

```ts
export function createWindowsCredentialKeyStore(opts?: {
  ffi?: WincredFFI;
}): KeyStore
```

The `WincredFFI` interface (in `src/keystores/types.ts`, see below)
abstracts the four FFI calls so tests inject a fake. Real impl uses
`bun:ffi` to dlopen `advapi32.dll` and bind:

```c
BOOL CredWriteW(
  PCREDENTIALW Credential,
  DWORD        Flags          // 0
);
BOOL CredReadW(
  LPCWSTR      TargetName,
  DWORD        Type,           // CRED_TYPE_GENERIC = 1
  DWORD        Flags,          // 0
  PCREDENTIALW *Credential
);
BOOL CredDeleteW(
  LPCWSTR      TargetName,
  DWORD        Type,
  DWORD        Flags
);
VOID CredFree(
  PVOID        Buffer
);
```

`CredWriteW` takes a `CREDENTIALW` struct by pointer. The struct
layout is fixed; we pack it as a TypedArray + DataView (offsets are
public ABI, stable since Windows 2000). Wide strings are UTF-16LE
with a NUL terminator.

### 4.2 Target-name namespacing

Use `Subscribetome:<ref>` (mirrors the Linux Secret Service
`subscribetome` service attribute). Both `Credential Manager` (the
control panel UI) and `cmdkey /list` will show entries under that
prefix.

### 4.3 Secret-not-in-argv guarantee

Critical posture detail: secret values must reach `CredWriteW`'s
`CredentialBlob` field directly through FFI — they must NEVER appear
in a process argv. Native FFI satisfies this trivially (the secret
lives in a typed-array we own and pass as a pointer). The macOS
backend's known `security -w <value>` limitation does NOT apply on
Windows — this is a strict posture improvement, like the Linux
stdin-pipe was.

### 4.4 New entries in `src/keystores/index.ts`

Add the win32 platform branch + alias names:

```ts
if (platform === "win32") {
  if (!isWincredReachable({ ffi: opts.ffi })) {
    cache = createUnsupportedKeyStore(
      "Windows Credential Manager is unreachable. " +
      "This usually means advapi32.dll could not be loaded — " +
      "are you running stm in a restricted sandbox?"
    );
    return cache;
  }
  cache = createWindowsCredentialKeyStore({ ffi: opts.ffi });
  return cache;
}
```

Plus aliases in `byName(name, opts)`:

```ts
case "windows":
case "windows-credential":
case "wincred":
case "credential-manager":
  return createWindowsCredentialKeyStore({ ffi: opts.ffi });
```

`isWincredReachable` is a cheap probe — call `CredReadW` with a
known-bogus name and check the error code. `1168 (ERROR_NOT_FOUND)`
means "API works, name missing"; `126 (ERROR_MOD_NOT_FOUND)` means
"advapi32 missing" → unsupported. Mirrors `probeLinuxSecretService`.

### 4.5 Types: `src/keystores/types.ts`

Add the FFI surface:

```ts
export interface WincredFFI {
  credWriteW(targetName: string, blob: Uint8Array, userName: string | null): boolean;
  credReadW(targetName: string): Uint8Array | null;       // null = ERROR_NOT_FOUND
  credDeleteW(targetName: string): boolean;
  lastError(): number;
}
```

Backend uses ONLY this interface — the real bun:ffi binding becomes a
swap-in implementation. Tests pass a recording stub.

### 4.6 Tests: `test/keystores-windows.test.ts`

Mirror `test/keystores.test.ts` for the macOS + Linux backends:

| Test | Assertion |
|---|---|
| `set` calls `credWriteW` with correct target name + blob | argv-shape equivalent: the value bytes match what we set |
| `set` throws on `lastError() != 0` | error message includes the Windows error code |
| `get` returns the value bytes UTF-8-decoded | round-trip with `set` works |
| `get` returns null when `credReadW` returns null | ERROR_NOT_FOUND = "missing" not "broken" |
| `delete` is idempotent (ERROR_NOT_FOUND on delete is OK) | matches Linux backend semantics |
| Resolver picks WindowsCredential on `platform === "win32"` with reachable FFI | analogous to the existing macOS / Linux platform-default tests |
| Resolver returns unsupported with friendly message when `advapi32.dll` is unloadable | the spec's "never silently fall back" invariant |
| `STM_KEYSTORE=wincred` override works from any platform | symmetric with the existing override tests |

CRITICAL ASSERTION: secret value never appears in any argv-shaped
buffer the FFI sees. The Windows backend's load-bearing posture
improvement over macOS.

### 4.7 Resolver cache discipline

Tests must call `_resetKeyStoreCache()` in `afterAll`. The macOS test
suite already follows this pattern — Windows tests need to do the same
so a forced `win32` platform doesn't leak into `import.test.ts` and
friends.

## 7. Risks + open questions

| Risk | Mitigation |
|---|---|
| `CREDENTIALW` struct layout drift across Windows builds | Layout is documented + stable since Windows 2000. We rely on documented public ABI. If a future Windows breaks it, our tests catch it on CI; users get a clear "Windows Credential Manager write failed: <code>". |
| Bun FFI on Windows is less battle-tested than on macOS/Linux | Probe at resolver init (`isWincredReachable`) — if the bindings fail to load, we return `unsupported (...)`, never silently fall back. The launcher banner / dashboard pill / `stm status` all surface the resolver's verdict. |
| WSL detection: WSL processes see neither Credential Manager nor a Linux keyring | Already handled at the resolver level by `process.platform === "linux"` inside WSL. The resolver returns the Linux Secret Service path (which will fail-explicit if D-Bus is unreachable). WSL→Windows interop via `/mnt/c/Windows/System32/...` is tracked separately in spec §5; out of scope for v0.5.0. |
| User running `stm` from inside a Windows sandbox / Hyper-V container without credential UI access | The probe catches this; error message tells the user. |

## 9. NOT in scope for v0.5.0

- WSL routing (treat as its own ship; spec §7 #5).
- A Windows-side hook-script template (Claude Code on Windows uses
  PowerShell or batch wrappers; `hooks/` shipscripts are bash today
  and they work in Git Bash / WSL. A `hooks/windows/` mirror with
  PowerShell wrappers can come later or be deferred indefinitely if
  the bash scripts run under Claude Code's Windows shell — verify
  empirically before deciding.)
- Bun FFI write-without-argv-exposure for the **macOS** Keychain. The
  spec calls this out as a v1.5 follow-up; bundling it here would mix
  concerns. Track it separately as a v0.5.x patch.
