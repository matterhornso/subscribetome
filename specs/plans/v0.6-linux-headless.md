# Plan — Workstream A tiers 2 + 3: Linux headless fallbacks

**Target version:** v0.6.0
**Spec parent:** [`specs/cross-platform-and-codex.md`](../cross-platform-and-codex.md) §5 (Linux row, Mitigation list)
**Status:** planned, unstarted
**Estimated build:** ~1 day. Two new backends + the tiered-resolver logic + the interactive passphrase UX is the fiddliest piece of work in the whole spec — spec §7 #3 calls it out explicitly.

## 1. Goal

`stm` works on Linux hosts where the Secret Service is unreachable:
SSH sessions, devcontainers, WSL, headless CI boxes, server installs
without a desktop. Today (v0.3.1+) those hosts get an honest
`unsupported (libsecret is installed but no Secret Service is
reachable…)` error — which is the *right* posture, but it's a wall.

After v0.6.0, `stm` falls through the chain `LinuxSecretService →
LinuxPass → EncryptedFile` and lands on the highest tier that works.
**Critical**: degradation is always visible. Spec §5 calls out the
gh CLI as the cautionary tale — gh silently falls back to plaintext
and we will not.

## 2. The tiers

| Tier | Backend | Requires | Works on |
|---|---|---|---|
| 1 | `LinuxSecretService` | libsecret + D-Bus session bus + unlocked keyring daemon | Linux desktop (GNOME, KDE with shim) |
| 2 | `LinuxPass` (NEW) | `pass(1)` + GPG agent | SSH sessions where the user already uses `pass` for password management |
| 3 | `EncryptedFile` (NEW) | nothing — derives from a user passphrase | Any host, last resort |

The resolver picks the **highest tier that works**, announces it
(`stm status`, dashboard, `LinuxPass (pass + GPG)` / `EncryptedFile
(0600, Argon2id)`), and an `$STM_KEYSTORE` override pins one
explicitly. An *insecure or weaker* backend is never selected silently.

## 3. Why this order, exactly

- **`LinuxPass` over `EncryptedFile`** because `pass` is the de
  facto standard headless secret manager on Linux. It's GPG-backed
  (the keyring already in the user's `gpg-agent`), works over SSH
  with agent forwarding, and is in every distro's repos.
- **`EncryptedFile` as the last resort** because it requires no
  external tools — every host has a filesystem and a passphrase. We
  own the cryptography; we eat the UX cost (passphrase prompts).
- **No third tier above Pass** like "Encrypted-only-with-a-yubikey"
  or "TPM-backed". Out of scope: v0.6.0 closes the gap, doesn't
  build a credentials product.

## 4. New files

### 4.1 `src/keystores/linux-pass.ts` — Tier 2

`pass` is a shell script over `gpg`. Layout: each secret is one
file under `$PASSWORD_STORE_DIR` (default `~/.password-store/`),
encrypted to the user's GPG key. The CLI:

```
pass insert -m subscribetome/<ref>        # secret on stdin
pass show       subscribetome/<ref>        # prints to stdout
pass rm         -f subscribetome/<ref>
```

Backend shape mirrors `linux-secret-service.ts`:

```ts
export function createLinuxPassKeyStore(opts?: {
  spawn?: SpawnFn;
}): KeyStore
```

**Posture detail**: `pass insert` reads the secret from stdin (with
`--multiline`, or via `printf "<value>\n<value>\n" | pass insert`
which the script accepts). Confirmed by reading `pass` source. NEVER
pass `--echo-passwords-without-confirmation` or any flag that puts the
secret on argv.

**Probe**: `probeLinuxPass({spawn})` runs `pass version` — exit 0 ⇒
installed; check for a configured GPG key by running `pass ls` and
inspecting the exit code (1 = no store yet, OK; 127 = pass missing;
gpg errors visible in stderr).

### 4.2 `src/keystores/encrypted-file.ts` — Tier 3

The pure-software fallback. One file under
`$XDG_DATA_HOME/subscribetome/keys.enc` (defaults to
`~/.local/share/subscribetome/keys.enc`). Mode 0600.

**Crypto** (no new runtime dependency — Bun has WebCrypto):

- **Key derivation**: Argon2id from the user's passphrase + a
  per-file 16-byte random salt. Argon2id is available in Bun via the
  built-in `Bun.password.hash()` and `verify` API (which uses Argon2
  internally) — BUT we need the raw 32-byte derived key for AES, so
  we shell out to `argon2` if available OR use a small JS Argon2id
  implementation (e.g. inline a vetted ~300-line port; we're already
  zero-deps for runtime).
  - **Alternative**: skip Argon2 and use PBKDF2-SHA512 with 600,000
    iterations via WebCrypto's `crypto.subtle.deriveBits` — already
    available in Bun, no port needed. Tradeoff: PBKDF2 is weaker
    against GPU attackers than Argon2id but is OWASP-2025-current at
    600k iterations and has no implementation risk. **Recommendation**:
    ship PBKDF2-SHA512 first as a v0.6.0 baseline; track Argon2id as
    a v0.6.1 patch once we pick a vendored implementation.
- **Cipher**: AES-256-GCM via WebCrypto, 12-byte IV, 16-byte tag.
- **File layout** (binary):
  ```
  magic (8 bytes) = "stmenc01"
  kdf_id (1 byte)  = 1 (PBKDF2-SHA512) or 2 (Argon2id, future)
  salt   (16 bytes)
  iv     (12 bytes)
  ciphertext (...)
  tag    (last 16 bytes of ciphertext)
  ```
  A JSON-serialized `{ ref: value }` map is the plaintext. Compact
  format keeps the file small (each entry is a few hundred bytes).
- **Concurrency**: file is loaded into memory once per backend
  instance; mutations write the whole file back atomically (write
  to `keys.enc.tmp`, rename). Lock file optional (we're single-user
  local).

**Passphrase UX** — spec §7 #3 explicitly notes this is the
fiddliest part:

- First write: prompt the user via stderr for a new passphrase
  (confirm twice). Cache in memory for the process lifetime.
- Subsequent reads: prompt for the passphrase on first read,
  cache in memory.
- Non-interactive (stdin not a TTY) reads: try `$STM_FILE_PASSPHRASE`
  env var first. If unset, the get returns null + the resolver
  reports a structured error so the hook can fail-safe (PreToolUse
  exits 0 without rewriting; the command runs with placeholder
  intact and fails harmlessly).
- A `stm vault unlock` CLI subcommand pre-warms an in-memory cache
  shared by the daemon (since the daemon already outlives a session,
  one unlock prompt covers many).

```ts
export function createEncryptedFileKeyStore(opts?: {
  filePath?: string;            // default XDG path
  passphraseProvider?: () => string;
  now?: () => number;
}): KeyStore
```

### 4.3 Resolver update — `src/keystores/index.ts`

The Linux branch becomes:

```ts
if (platform === "linux") {
  // Tier 1: Secret Service
  if (which("secret-tool") && probeLinuxSecretService({ spawn: opts.spawn })) {
    cache = createLinuxSecretServiceKeyStore({ spawn: opts.spawn });
    return cache;
  }
  // Tier 2: pass(1)
  if (which("pass") && probeLinuxPass({ spawn: opts.spawn })) {
    cache = createLinuxPassKeyStore({ spawn: opts.spawn });
    return cache;
  }
  // Tier 3: encrypted file. Always works, but interactive: needs
  // a passphrase. The resolver gates on `STM_ALLOW_FILE_BACKEND=1`
  // for first-touch so a fresh install doesn't silently create an
  // encrypted file the user didn't ask for.
  if (env.STM_ALLOW_FILE_BACKEND === "1" || encryptedFileExists()) {
    cache = createEncryptedFileKeyStore();
    return cache;
  }
  cache = createUnsupportedKeyStore(
    "No keystore is reachable on this host:\n" +
      "  · secret-tool: " + (which("secret-tool") ? "installed but no Secret Service" : "not installed") + "\n" +
      "  · pass(1):     " + (which("pass") ? "installed but no usable GPG store" : "not installed") + "\n" +
      "  · encrypted-file fallback: gated behind STM_ALLOW_FILE_BACKEND=1 (opt-in)\n" +
      "Run `stm doctor` for the full diagnosis.",
  );
  return cache;
}
```

**Critical**: the encrypted-file fallback is **opt-in via env var on
first touch**. The user explicitly authorizes the weaker tier.
Subsequent runs auto-use it (the file existing IS the consent).

### 4.4 New CLI subcommand: `stm doctor`

`stm doctor` runs the full resolver probe and prints a structured
diagnosis: which tier is active, which tiers were skipped + why,
how to upgrade to a stronger tier. Reuses the `doctor()` pattern
from `codex-hooks.ts`.

```
keystore tiers (Linux):
  ✓ Tier 3 — EncryptedFile (active)
    ~/.local/share/subscribetome/keys.enc, 0600

  ✗ Tier 2 — LinuxPass (pass + GPG)
    pass is on PATH but no GPG key configured.
    Fix: `gpg --quick-generate-key your@email.com default default 1y`
         then `pass init your@email.com`.
    Then re-run `stm doctor` to confirm Tier 2 is reachable.

  ✗ Tier 1 — LinuxSecretService (libsecret)
    secret-tool is not installed.
    Fix: `apt install libsecret-tools` (Debian/Ubuntu).
    This is the strongest tier on Linux desktops.
```

## 5. Tests

Three new files:

### 5.1 `test/keystores-linux-pass.test.ts`

| Test | Assertion |
|---|---|
| `set` calls `pass insert -m subscribetome/<ref>` with secret via stdin | secret NOT in argv |
| `get` parses `pass show` stdout, strips trailing newline | matches the Linux SS pattern |
| `delete` calls `pass rm -f subscribetome/<ref>` | idempotent |
| `probeLinuxPass` returns true when `pass version` exits 0 + a store exists | passes the "fully usable" bar |
| `probeLinuxPass` returns false when pass is installed but no GPG key configured | catches the common "installed but useless" state |

### 5.2 `test/keystores-encrypted-file.test.ts`

| Test | Assertion |
|---|---|
| Round-trip: set + get returns the same value | core crypto correctness |
| Set + delete + get returns null | delete works |
| Wrong passphrase on read throws a clear error (not silent corruption) | GCM tag detects tampering |
| File magic + version are written + checked | older versions reject gracefully |
| File is written with mode 0600 | filesystem ACL is part of the security story |
| Atomic write: corrupting the file mid-write doesn't lose data | tmpfile + rename pattern |
| Passphrase cached in-memory for the process lifetime | one prompt per session, not per call |
| Non-TTY without STM_FILE_PASSPHRASE returns null + structured error | hooks fail safe (per the spec's PreToolUse invariant) |

### 5.3 `test/keystores-tier-resolver.test.ts`

| Test | Assertion |
|---|---|
| Resolver picks Tier 1 when libsecret + D-Bus reachable | tier order |
| Resolver picks Tier 2 when libsecret fails BUT pass + GPG OK | falls through correctly |
| Resolver picks Tier 3 ONLY when STM_ALLOW_FILE_BACKEND=1 (first touch) | opt-in invariant |
| Once the encrypted file exists, Tier 3 is auto-selected on subsequent calls | "file existing IS consent" |
| Headless Linux without any tier returns `unsupported (...)` with all three reasons | the gh-cli-cautionary-tale invariant |
| `STM_KEYSTORE=linux-pass` override pins Tier 2 even when Tier 1 is reachable | symmetric with existing override tests |
| `STM_KEYSTORE=encrypted-file` works | opt-in is implicit in the explicit override |

## 8. Risks + open questions

| Risk | Mitigation |
|---|---|
| Passphrase UX over SSH — TTY detection works differently in containers, mosh, screen, tmux | Make `STM_FILE_PASSPHRASE` the canonical non-interactive route; document `stm vault unlock` for interactive pre-warming via the long-lived daemon. |
| Argon2id implementation risk if we vendor a port | Ship PBKDF2-SHA512 in v0.6.0; Argon2id behind a feature flag in v0.6.1 after we settle on a vendored impl. PBKDF2 at 600k is OWASP-2025-recommended. |
| Encrypted-file backend creates a forever-stuck-with-this-passphrase situation | Add `stm vault rotate-passphrase` in the same release. Decrypt-then-re-encrypt under a new key. Backed up to `.bak`. |
| `pass init` not having been run + user doesn't realize Tier 2 is one step away | `stm doctor` makes the gap explicit and prints the exact `gpg` + `pass init` commands. |
| WSL routing — WSL processes detected as `platform: 'linux'` and would default-fall through to EncryptedFile | Document a v0.6.1 follow-up: detect WSL via `/proc/version` and either route to a Windows credential-helper `.exe` (spec §5 WSL row) or to EncryptedFile with a clear "WSL detected — using encrypted-file fallback" message. |

## 10. NOT in scope for v0.6.0

- WSL-specific routing (treat as v0.6.1, after the Windows backend
  lands so WSL→Windows-interop is feasible).
- Hardware-token tiers (TPM, YubiKey via PIV). Far outside the
  v1 scope.
- A cross-machine sync mode for EncryptedFile. v1 is local-only;
  spec §1 explicit non-goal.
- Argon2id (v0.6.1 — PBKDF2-SHA512 ships first for zero
  implementation risk).
